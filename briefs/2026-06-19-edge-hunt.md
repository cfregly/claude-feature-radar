# Edge hunt, 2026-06-19

A full-surface re-sweep for new Claude edges against OpenAI and Gemini, with every candidate taken to
a LIVE acceptance probe or a LIVE head-to-head on all three keys. Internal both-directions record.
The public surface (README, edge bundles) carries only the verified wins.

## Method

1. Refreshed every source in `engine/sources_registry.py` with `make edges` (the $0 stdlib sweep).
2. Fanned out four subfeature-depth analysts over the candidate surface (agentic control plane,
   reasoning and output-quality knobs, throughput and economics, grounding and documents), each
   comparing Claude against the nearest OpenAI and Gemini subfeature on the live docs.
3. Ran a live acceptance probe on this org's key for each top candidate before any benchmark.
4. Built demonstrators and ran live head-to-head benchmarks for the survivors on all three keys.

## Shipped this pass (verified Claude wins, all arms ran)

### pdf-citations (demoKind pdf_grounding) -> claude-ahead

Workload: five questions over a five-page synthetic agreement PDF supplied directly in the request.
The gate is a verifiable per-page pointer into that PDF, checked against the known answer page.

| arm | answered | direct-PDF pointer | right page | cost |
|---|:---:|:---:|:---:|---:|
| Claude Haiku 4.5, inline PDF + citations | 5/5 | 5/5 | 5/5 | $0.0458 |
| OpenAI GPT-5.4, input_file | 5/5 | 0/5 | 0/5 | $0.0064 |
| Gemini 3.5 Flash, inline PDF | 5/5 | 0/5 | 0/5 | $0.0322 |

All three read the PDF and answered correctly. Only Claude returned a verifiable page pointer into the
directly-supplied PDF. Scoped to the direct-PDF path: the competitors can cite through a hosted
file-search store (a different, persisted flow). OpenAI was cheaper here, a cost negative kept off the
public surface. The edge is grounding fidelity. Receipt: `edges/pdf-citations/receipt.json`.

### search-results (demoKind byo_rag_grounding) -> claude-ahead

Workload: five questions over five developer-supplied RAG chunks. The gate is a citation that resolves
to the correct source chunk and the hosted-store objects required to get it.

| arm | correct cite | pointer kind | hosted objects | cost |
|---|:---:|:---:|:---:|---:|
| Claude Haiku 4.5, inline search_result blocks | 5/5 | block-span | 0 | $0.0067 |
| OpenAI GPT-5.4, hosted file_search | 5/5 | file-level | 6 | $0.0300 |
| Gemini 3.5 Flash, hosted file_search | 5/5 | chunk-level | 6 | $0.0146 |

All three cited the correct source. Claude did it inline, resolver-free, block-level, zero persisted
objects, while the competitors required a hosted vector store and returned a coarser pointer. The edge is
the inline, no-store, block-level mechanism, not "only Claude can cite." Receipt:
`edges/search-results/receipt.json`.

## Confirmed live, novel, but HELD (value did not reproduce on this pass)

### advisor_tool (demoKind advisor_routing)

The mechanism is real and unmatched: a cheap executor model calls a frontier advisor model as a
server-side sub-inference inside ONE `/v1/messages` request, billed separately in `usage.iterations`.
Probed live and accepted (Sonnet executor + Opus advisor, beta `advisor-tool-2026-03-01`). Neither
OpenAI (Agents SDK orchestration is client-side separate calls) nor Gemini (Interactions multi-step
tool use) documents a single-request stronger-model sub-inference.

The cost-at-quality WIN did not reproduce on the built slice (`make advisor`): on eight small
single-shot coding tasks every arm scored 8/8, so there was nothing for the advisor to rescue, and the
advisor arm was the MOST expensive ($0.3265 vs Opus-solo $0.0568) because you pay the Opus planning
pass plus the executor on every task. This is the "measure where it bites" trap, caught by the gate.
The advisor's savings come from the executor generating the bulk output at the cheaper rate, so the
edge needs a long-output / long-horizon workload where that bulk is large and a weak executor genuinely
fails without the plan. Held within-Claude / never-proven cross-vendor until that workload is built.
The demonstrator and the receipt (`data/last_advisor.json`) stay in the repo as the measurement tool.

### batch_processing extended output (300k, beta output-300k-2026-03-24)

Documented capability gap: a single batch turn can emit up to 300k output tokens on Claude vs 128k on
GPT-5.5 and ~65k on Gemini 3.5 Flash, at the same 50% batch discount. Not benchmarked live this pass
(a 200k-output batch generation is slow and a few dollars). Held as a documented gap pending a live
single-deliverable run that genuinely exceeds 128k.

## Killed on the live cross-vendor check (do not pitch)

- **structured_outputs + server tools**: the hypothesis was that Claude could compose strict
  constrained-decoding JSON with a server tool where OpenAI could not. Probed live: Claude accepted
  strict `output_config.format` + `code_execution` (valid JSON), AND OpenAI accepted strict
  `text.format` json_schema + `web_search` in one Responses call (valid JSON). Parity. Claude is also
  behind on recursive `$ref` schemas (OpenAI and Gemini support root recursion, which Claude lists as
  unsupported). Not an edge.
- **adaptive_thinking**: parity. OpenAI (gpt-5.5 interleaved + adaptive reasoning) and Gemini (dynamic
  thinking by default, thinkingBudget -1) both ship model-decided depth interleaved between tool calls.
- **effort**: parity on the headline. OpenAI `reasoning.effort` matches through `xhigh` and adds a
  `none` rung below Claude's `low`, and Gemini caps at `high`. Claude's `max` is the only rung above
  `xhigh`, and the docs warn it adds cost for small gains. The only narrow distinction (one dial over
  all output tokens incl. tool calls with thinking off) did not rise to a measured founder-value win.
- **tool_search**: parity with OpenAI (both ship `defer_loading` with the same cache-preserving
  end-of-context injection and ~85% definition-token reduction). A Gemini-only gap, but the rule is
  best-to-best, so this is parity.
- **fine_grained_tool_streaming**: a DX / first-byte-latency feature with an intrinsic invalid-JSON
  caveat, not a founder-priced axis.
- **mid_conversation_system**: parity vs OpenAI's `developer` role on the priority axis. The only
  possible wedge (cache-preserving mid-loop injection) hinges on an undocumented OpenAI caching detail
  and is clean only vs Gemini. Not pitched.

## What stands after this pass

Shipped Claude-ahead edges from this pass: programmatic-tool-calling, citations, exact-list-ledger,
cache-diagnostics, task-budgets, pdf-citations, search-results. Later passes the same day promoted three
more (see the combination-edges and deep-dive briefs): grounding-stack, web-citations, and
bulk-extended-output, which is the batch-300k gap proven by a 230,607-token single batch turn. Held with
reason: advisor_tool (novel, value not yet reproduced) and dynamic-web-filtering. Killed/parity:
the six above. The engine refused to ship the advisor edge and the structured-outputs edge on the live
check, which is the honesty posture working as designed.
